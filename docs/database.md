## Indexes

A focused set, every one of them justified by a query the application
actually runs.

| Table | Index | Type | Purpose |
|---|---|---|---|
| `Sales` | `IX_Sales_SaleNumber` | btree UNIQUE | enforces business-id uniqueness; backs `SaleNumberExistsAsync` |
| `Sales` | `IX_Sales_CustomerId_SaleDate` | btree composite | "sales for customer X by date" listing path |
| `Sales` | `IX_Sales_BranchId_SaleDate` | btree composite | "sales for branch X by date" listing path |
| `Sales` | `IX_Sales_SaleDate_Active` | btree **partial** `WHERE IsCancelled = false` | the common-case list query stays tight without bloating with cancelled rows |
| `Sales` | `IX_Sales_SaleDate_Cancelled` | btree **partial** `WHERE IsCancelled = true` | the minority case keeps the same access pattern, cheap |
| `Sales` | `IX_Sales_SaleNumber_Trgm` | **GIN** `gin_trgm_ops` | substring filter `saleNumber=*foo*` uses the index instead of seq-scanning |
| `SaleItems` | `UX_SaleItems_SaleId_ProductId_Active` | btree **partial** UNIQUE `WHERE IsCancelled = false` | encodes the aggregate's per-product invariant at the DB level вҖ” cancelled rows are exempt so re-add after a cancel is allowed |
| `Users` | `IX_Users_Email` / `IX_Users_Username` | btree UNIQUE | authentication path; unique business identifiers |
| `OutboxMessages` | `IX_OutboxMessages_Pending` | btree **partial** `WHERE ProcessedAt IS NULL` | the dispatcher's hot-path query (`ORDER BY OccurredAt LIMIT 50`) walks only pending rows |

Why partials? Postgres skips index entries that fail the filter, so
the index pages stay small even after months of dispatched rows or
cancelled sales accumulate. A regular btree on `OutboxMessages.OccurredAt`
would balloon вҖ” pending rows are seconds old; dispatched rows live
until cleanup runs.

## Triggers

### `trg_sales_bump_rowversion` вҖ” BEFORE UPDATE on `Sales`

Maintains `Sales.RowVersion` for optimistic concurrency. Migration:
[20260510213727_ReplaceXminWithRowVersionTrigger.cs](../template/backend/src/Ambev.DeveloperEvaluation.ORM/Migrations/20260510213727_ReplaceXminWithRowVersionTrigger.cs).

```sql
CREATE OR REPLACE FUNCTION ambev_sales_bump_rowversion()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
    NEW."RowVersion" := COALESCE(OLD."RowVersion", 0) + 1;
    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_sales_bump_rowversion
BEFORE UPDATE ON "Sales"
FOR EACH ROW
EXECUTE FUNCTION ambev_sales_bump_rowversion();
```

EF Core's mapping marks the column as
`ValueGeneratedOnAddOrUpdate + IsConcurrencyToken`, so:

- EF emits `UPDATE вҖҰ WHERE Id = @id AND RowVersion = @oldVersion`
- The trigger bumps the value, RETURNING brings it back
- A zero-row update produces `DbUpdateConcurrencyException` вҶ’ 409

Why a trigger and not `xmin`? `xmin` (the original implementation)
gets reset by `VACUUM FREEZE` when its age crosses
`vacuum_freeze_table_age` (default 150 M txns). That would
invalidate every cached ETag at once. The bigint version is monotonic
across the row's lifetime and immune to FREEZE.

### `trg_outbox_notify_pending` вҖ” AFTER INSERT on `OutboxMessages`

Statement-level (one notification per batch, not per row). Migration:
[20260510221545_OutboxNotifyTrigger.cs](../template/backend/src/Ambev.DeveloperEvaluation.ORM/Migrations/20260510221545_OutboxNotifyTrigger.cs).

```sql
CREATE TRIGGER trg_outbox_notify_pending
AFTER INSERT ON "OutboxMessages"
FOR EACH STATEMENT
EXECUTE FUNCTION ambev_outbox_notify_pending();
```

The function calls `pg_notify('outbox_pending', '')`. The dispatcher
keeps a dedicated `LISTEN outbox_pending` connection open and wakes up
within milliseconds вҖ” publish latency drops from "up to 5 s" (the
poll interval) to sub-second under load.

## Connection tuning

Connection string for the WebApi вҶ’ DB pool (set via
`ConnectionStrings:DefaultConnection`; the dev `.env` ships a
representative value):

```
Host=...;Port=5432;Database=...;Username=...;Password=...;
Pooling=true;
Minimum Pool Size=10;
Maximum Pool Size=200;
Keepalive=30;
```

EF Core configuration on top
([Program.cs:203](../template/backend/src/Ambev.DeveloperEvaluation.WebApi/Program.cs#L203)):

- `AddDbContextPool<DefaultContext>(..., poolSize: 256)` вҖ” the EF Core
  context pool recycles the compiled model + change tracker across
  requests. Saves per-request allocations; the **connection** pool
  cap above is the real concurrency ceiling.
- `EnableRetryOnFailure(3, 2s)` вҖ” survives transient Postgres
  disconnects (PG failover, network blip) without surfacing them.
- `.CommandTimeout(15)` вҖ” caps any runaway query at 15 s. Misses
  surface as `OperationCanceledException` and become 500s in logs.
- `EnableSensitiveDataLogging(false)` вҖ” never leak query parameters
  to logs in production.

## Performance choices worth calling out

- **Compiled queries**. The two hottest reads use
  `EF.CompileAsyncQuery`: `Sale.GetByIdAsync` (touched by every
  write path) and `User.GetByEmailAsync` (every authn request). See
  [`SaleRepository.cs`](../template/backend/src/Ambev.DeveloperEvaluation.ORM/Repositories/SaleRepository.cs),
  [`UserRepository.cs`](../template/backend/src/Ambev.DeveloperEvaluation.ORM/Repositories/UserRepository.cs).
- **No-tracking pre-checks**. `SaleNumberExistsAsync` /
  `EmailExistsAsync` use `AsNoTracking() + AnyAsync()` вҖ” no rows
  materialised. The unique index remains the source of truth, so a
  concurrent insert that slips past returns 409 via the exception
  middleware.
- **Keyset pagination available** for high-scale listing вҖ” O(log n)
  per page, no COUNT(\*). See [api.md вҶ’ List sales](api.md#get-apiv1sales--list-sales).
- **Two-phase outbox dispatch** keeps row locks short even when the
  publish call is slow. See
  [architecture.md вҶ’ Dispatcher mechanics](architecture.md#dispatcher-mechanics).
- **Outbox cleanup runs in 5 000-row chunks**, each in its own short
  transaction вҖ” autovacuum keeps up and WAL doesn't spike. Source:
  [`OutboxCleanupService.cs`](../template/backend/src/Ambev.DeveloperEvaluation.ORM/Outbox/OutboxCleanupService.cs).

## Known future work

Documented gaps that did not cross the cost/benefit line for this
submission, all reachable from the application's current state:

   Cleanup becomes `DROP PARTITION` instead of a chunked DELETE вҖ”
   instantaneous, no autovacuum churn.

   (CTE-based batched UPDATE instead of per-row).
4. **2nd-level cache for `ListSales`** with cache-aside invalidation
   on any sale write. Today only the by-id read is cached.

## Auto-migration & multi-pod rollouts

The application **does not** call `Database.MigrateAsync()` at startup.
Migrations are applied via `dotnet ef database update` at deploy time,
which is also what the integration tests do against the testcontainer.

If a future iteration calls `MigrateAsync()` on startup, wrap it in a
Postgres advisory lock so concurrent rolling-deploy pods do not race:

```csharp
await using (var conn = new NpgsqlConnection(connString))
{
    await conn.OpenAsync();
    await using (var cmd = conn.CreateCommand())
    {
        cmd.CommandText = "SELECT pg_advisory_lock(8945312094);";
        await cmd.ExecuteNonQueryAsync();
    }
    await context.Database.MigrateAsync();
    // Advisory lock released by session close (the `await using` above).
}
```

Exactly one pod runs `MigrateAsync()`; the rest block cooperatively,
see the schema up-to-date, and return cleanly.
